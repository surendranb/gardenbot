# Hello World: The Origins of GardenOS

GardenOS wasn't built to be a simple "sensor box." It was built to be a **Curious Warden**—an agent that doesn't just read numbers, but *investigates* the biome using vision and deterministic math. This is the story of how it's set up and how it thinks.

<!-- more -->

## 🛠 The Hardware Stack
We've kept the hardware lean and focused on "Local Truth." The system is entirely local-first, avoiding cloud-dependency where possible.

*   **Brain**: An old MacBook Air running the scripts and OpenClaw.
*   **Controller**: An **Arduino** acting as the serial bridge for the sensor array.
*   **Atmosphere**: A **DHT11** sensor for temperature and humidity.
*   **Light**: A dedicated **Lux sensor** to track solar exposure.
*   **Soil Moisture**: Three **Capacitive Moisture Sensors** (p1: Nickels, p2: Mint, p3: Pothos).
*   **Vision**: A standard **Webcam** mounted for top-down biome surveillance.

## 🧠 The Logical Flow

The system runs as a pipeline of independent scripts, not one monolithic process. Here's what actually happens each cycle:

1.  **System-level triggers**: `warden.py`, `vision.py`, and `weather_scout.py` run on a schedule via **cron/launchd** on the MacBook. They're plain Python scripts — no framework orchestrates them.
2.  **Sensor collection**: `warden.py` connects to the Arduino over serial, reads temperature, humidity, light, and soil moisture, and writes the readings to `telemetry.csv`, `metrics.csv`, `current_snapshot.json`, and `warden_state.json`.
3.  **Visual interpretation**: `vision.py` grabs a frame from the webcam via OpenCV, then sends it to **Gemma 3 on Google AI Studio**. Gemma 3 describes what it sees — leaf color, posture, soil surface condition — and the output lands in `vision_observation.json` and `vision_observation.md`. This is perception, not reasoning.
4.  **Weather context**: `weather_scout.py` fetches the current outdoor Chennai weather from OpenWeatherMap. This gives us the macro-context — what the outdoor environment is doing vs. what's actually happening on the desk.
5.  **Context synthesis (SILICA)**: `prep_observer_context.py` pulls together all the data files, merges them with the World Model (`GARDEN_MANIFEST.md`) and plant config (`plants.json`), and produces a single `observer_context.md`. This is where raw numbers become semantic facts.
6.  **Reasoning**: OpenClaw's Warden agent reads `observer_context.md` and sends it to an LLM. The model reasons about plant health — cross-checking sensor readings against the visual evidence, comparing against historical trends, and flagging anything that needs attention.
7.  **Reporting**: The Warden writes its findings to `vision_ledger.md` and posts a summary to Slack `#plantclaw`.
8.  **Publishing**: `sync.sh` builds the MkDocs site, commits everything to GitHub, and pushes to GitHub Pages. The live dashboard reads the CSVs directly from the repo.

The key design choice here is **decoupling**. Each script runs independently. If the LLM is down, sensors still log. If the webcam is unplugged, telemetry still records. If weather API is flaky, everything else keeps going.

## 🌿 Final Thoughts
It's not just a garden; it's a monitored biome.
